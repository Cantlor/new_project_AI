# DEVELOPMENT_WORKFLOW.md

## 1. Назначение документа

Этот документ фиксирует **рабочий процесс разработки** проекта **«ИИ для полей»**.

Его задача:

- задать единый порядок взятия задачи в работу;
- не дать разработке уйти в хаотичный code-first режим;
- связать повседневную работу с `main_tech.md`, модульными ТЗ, `DATA_CONTRACT.md`, `MANIFEST_SCHEMAS.md`, `REPO_CONVENTIONS.md`, `TESTING_STRATEGY.md`, `EXPERIMENT_TRACKING.md` и `DECISIONS.md`;
- сделать change set, проверку, handoff и ревью воспроизводимыми.

Этот документ не заменяет ТЗ модулей и не дублирует архитектурные решения.  
Он отвечает на вопрос: **как именно вести работу над задачей от постановки до handoff и review-ready состояния**.

---

## 2. Главный принцип

В проекте действует правило:

**сначала контракт и рамки изменения, потом код, потом проверка, потом handoff.**

Запрещённый путь:

- быстро написать «почти работающий» код;
- по ходу менять смысл данных;
- не зафиксировать новые manifests / summaries / runtime decisions;
- оставить следующему человеку разбираться, что именно было сделано.

---

## 3. Source of truth для повседневной работы

Перед началом любой задачи нужно понимать, какой документ является главным источником истины.

### 3.1. Порядок приоритета

1. архитектурный freeze / принятые решения из `DECISIONS.md`;
2. `main_tech.md`;
3. ТЗ соответствующего модуля;
4. `DATA_CONTRACT.md`;
5. `MANIFEST_SCHEMAS.md`;
6. `REPO_CONVENTIONS.md`;
7. `TESTING_STRATEGY.md`;
8. `EXPERIMENT_TRACKING.md`;
9. локальный README / комментарии в коде;
10. устные или чатовые договорённости, если они не перенесены в документы.

Если изменение конфликтует с верхним уровнем, код писать нельзя, пока конфликт не прояснён или не оформлен как новое решение.

---

## 4. Жизненный цикл задачи

Каждая задача должна проходить через одни и те же стадии:

```text
1. intake
2. scope lock
3. design check
4. implementation
5. validation
6. evidence capture
7. handoff / review
```

### 4.1. Intake

На входе нужно явно понять:

- **что именно нужно изменить**;
- **в каком модуле / слоях это изменение живёт**;
- **затрагивает ли оно data contract, manifests, runtime или только локальную реализацию**;
- **какой артефакт будет критерием успеха**.

### 4.2. Scope lock

До начала кода нужно зафиксировать:

- что входит в change set;
- что явно не входит;
- какие инварианты нельзя ломать;
- какие документы нужно перечитать перед изменением.

### 4.3. Design check

Перед кодом обязательно проверить:

- не конфликтует ли изменение с `valid`-policy;
- не ломает ли `feature_mode` / `assembled_model_input`;
- не требует ли новой записи в `DECISIONS.md`;
- не изменит ли оно format / schema manifest-ов;
- не меняет ли оно сравнимость run-ов.

### 4.4. Implementation

Код пишется только после design check.

### 4.5. Validation

После кода выполняются обязательные проверки.

### 4.6. Evidence capture

Нужно сохранить:

- что реально менялось;
- что реально проверялось;
- какие артефакты были сгенерированы;
- какой статус у результата.

### 4.7. Handoff / review

Задача должна завершаться не только кодом, но и понятной передачей контекста дальше.

## 5. Как брать задачу в работу

Перед началом любой задачи нужно ответить на 5 вопросов.

### 5.1. Какой это тип задачи?

В проекте различаются:

- **contract task** — изменение документов, контрактов, схем, соглашений;
- **implementation task** — изменение кода внутри уже принятого контракта;
- **integration task** — стык модулей;
- **experiment task** — запуск/сравнение run-ов;
- **forensic task** — разбор результатов, manifests, метрик, runtime поведения;
- **cleanup/refactor task** — улучшение структуры без изменения контракта.

### 5.2. Какой модуль является основным?

Нужно выбрать один главный scope:

- `module_prep_data`
- `module_net_train`
- `module_target_predict`
- `module_postprocess_vectorize`
- `module_eval`
- либо `global/docs`

### 5.3. Какие документы обязательны к чтению перед началом?

Минимум:

- `DATA_CONTRACT.md`
- `MANIFEST_SCHEMAS.md`
- ТЗ целевого модуля
- `REPO_CONVENTIONS.md`
- при необходимости `TESTING_STRATEGY.md` и `DECISIONS.md`

### 5.4. Какие инварианты нельзя ломать?

Во всех задачах по умолчанию нельзя ломать:

- канонический pipeline;
- `valid` как mask + input channel;
- `raw8/raw8_idx3` и `raw8_valid/raw8_idx3_valid` contract;
- manifest-first policy;
- checkpoint-driven predict;
- честную сравнимость run-ов.

### 5.5. Какой минимальный результат нужен?

Нужно заранее определить, что будет считаться успехом:

- исправленный validator;
- новый manifest writer;
- train run с checkpoint metadata;
- predict run с корректным `predict_manifest.json`;
- integration test;
- обновлённый handoff.

## 6. Как формировать change set

Изменение должно быть **минимально достаточным**.

### 6.1. Хороший change set

Хороший change set:

- решает одну связанную проблему;
- не тянет лишний рефактор «заодно»;
- не переименовывает массово сущности без причины;
- не смешивает contract change, refactor и experiment в один коммит/пакет;
- явно показывает, какие файлы изменены и почему.

### 6.2. Плохой change set

Плохой change set:

- одновременно меняет dataset contract, train loop и output manifests;
- включает несвязанные rename и stylistic edits;
- меняет логику и не обновляет документы;
- добавляет fallback-магии без явной фиксации.

### 6.3. Change classification

Перед реализацией change set нужно пометить как один из вариантов:

- `docs-only`
- `contract-impacting`
- `implementation-only`
- `integration-impacting`
- `experiment-only`
- `refactor-no-contract-change`

Если change — `contract-impacting`, сначала обновляются документы, а потом код.

## 7. Правила реализации

### 7.1. Писать снизу вверх

Обычный порядок внутри задачи:

1. schemas / config;
2. validators;
3. pure/core logic;
4. I/O and adapters;
5. stage runner / CLI;
6. manifests / summaries;
7. tests.

### 7.2. Не смешивать слои ответственности

Один файл не должен одновременно:

- парсить CLI;
- принимать runtime policy;
- делать core processing;
- писать outputs;
- писать manifests;
- решать compatibility fallback.

### 7.3. Не менять смысл данных тихо

Если меняется:

- `feature_mode`;
- `channel_semantics`;
- `valid` semantics;
- target encoding;
- normalization rules;
- output schema;

то это уже не «локальная реализация», а изменение контракта или как минимум интерфейса.

### 7.4. Явные ошибки лучше скрытой совместимости

При невозможности корректно интерпретировать вход или восстановить контракт система должна падать с явной ошибкой, а не пытаться молча «угадать» поведение.

## 8. Обязательный рабочий ритуал перед кодом

Перед тем как менять код, нужно пройти короткий pre-change checklist.

### 8.1. Pre-change checklist

- Прочитан релевантный модульный spec.
- Проверено, не конфликтует ли изменение с `DATA_CONTRACT.md`.
- Проверено, не затрагивается ли `MANIFEST_SCHEMAS.md`.
- Понято, нужно ли обновлять `DECISIONS.md`.
- Зафиксирован scope change set.
- Понято, какие тесты/валидации обязательны после изменения.
- Понято, какой handoff потом нужно оставить.

Если хотя бы на один пункт ответа нет, задача ещё не готова к реализации.

---

## 9. Как проверять изменение

После реализации нельзя ограничиваться только тем, что «скрипт отработал».

### 9.1. Минимальные проверки

В зависимости от задачи обязательны:

- unit tests для чистой логики;
- integration checks на стык модулей;
- manifest/schema validation;
- smoke run соответствующего entrypoint;
- ручная sanity-проверка артефактов там, где это критично.

### 9.2. Что обязательно проверить в проекте

Особое внимание всегда нужно уделять:

- `valid` semantics;
- `feature_channel_count` и `final_input_channel_count`;
- `channel_semantics`;
- spatial compatibility;
- наличию `config_used`, `manifest`, `summary`;
- достаточности checkpoint metadata;
- честности source provenance.

### 9.3. Negative checks

Если задача касается validators, compatibility или manifests, нужно проверять и ожидаемые ошибки, а не только happy-path.

---

## 10. Как собирать evidence после изменения

После выполнения задачи нужно зафиксировать доказательства результата.

### 10.1. Что сохранять

Минимум:

- список изменённых файлов;
- короткое описание, что именно изменено;
- список выполненных проверок;
- status результата;
- пути к главным артефактам;
- известные ограничения или риски.

### 10.2. Где это отражать

В зависимости от формата работы evidence должно быть отражено в одном или нескольких местах:

- `summary.json`
- `run_report.md`
- `handoff.md`
- комментарий к change / PR / review note

---

## 11. Когда обновлять manifests и summaries

### 11.1. Обновление обязательно, если изменение затронуло

- входные артефакты модуля;
- выходные артефакты модуля;
- resolved contract;
- runtime policy;
- source provenance;
- threshold / normalization / valid / AOI policy;
- training или predict metadata.

### 11.2. Обновление не обязательно, если изменение строго локальное

Например:

- исправление комментария;
- рефактор внутренней функции без смены поведения;
- улучшение лога без смены runtime semantics.

Но даже тогда нужно быть уверенным, что поведение действительно не изменилось.

---

## 12. Когда обновлять документацию

### 12.1. Нужно обновлять документы, если изменение

- меняет baseline behavior;
- меняет термин или canonical name;
- меняет data contract;
- меняет manifest schema;
- меняет порядок run artifacts;
- меняет fair comparison rules;
- закрывает или открывает архитектурный вопрос.

### 12.2. Какой документ обновлять

- новый термин -> `GLOSSARY.md`
- новый cross-module data rule -> `DATA_CONTRACT.md`
- новая schema / новое обязательное поле -> `MANIFEST_SCHEMAS.md`
- новое репозиторное правило -> `REPO_CONVENTIONS.md`
- новое testing правило -> `TESTING_STRATEGY.md`
- новое правило сравнения экспериментов -> `EXPERIMENT_TRACKING.md`
- новое архитектурное решение -> `DECISIONS.md`
- новый шаг реализации / новый порядок исполнения -> `IMPLEMENTATION_PLAN.md`

## 13. Когда задача считается готовой к review

Задача считается `review-ready`, только если одновременно выполнены все условия:

1. scope задачи не расползся;
2. код не противоречит действующим контрактам;
3. нужные tests/checks выполнены;
4. manifests / summaries / config_used сохранены, если это требовалось;
5. нет скрытых fallback-магий;
6. обновлены нужные документы, если изменение затронуло контракт;
7. есть понятный handoff / краткое описание результата.

Если хотя бы один пункт не выполнен, задача ещё не review-ready.

---

## 14. Когда нужен handoff

Handoff обязателен, если:

- работа не доведена до конца;
- задача переходит другому человеку / агенту;
- был выполнен experiment-like run;
- есть риск забыть важные ограничения, выводы или следующий шаг;
- change set сложнее, чем тривиальная локальная правка.

### 14.1. Что должно быть в handoff

Минимум:

- контекст задачи;
- что было целью;
- что реально сделано;
- что не сделано;
- какие файлы затронуты;
- какие проверки выполнены;
- какие риски остались;
- что делать следующим шагом.

Для этого использовать шаблон `handoff.template.md`.

## 15. Специальные правила для AI-assisted разработки

ИИ-ассистенты в проекте используются как ускоритель, но не как источник архитектурной истины.

### 15.1. Что можно делегировать ассистенту

- локальную реализацию по уже зафиксированному контракту;
- validators;
- serializers;
- manifest writers;
- CLI glue;
- targeted refactor;
- unit/integration tests;
- forensic review и hidden-fallback audit.

### 15.2. Что нельзя делегировать без дополнительной проверки

- изменение data contract;
- смену `valid` semantics;
- смену feature contract;
- переопределение target semantics;
- изменение fair comparison rules;
- скрытое изменение baseline architecture.

### 15.3. Как правильно ставить задачу ассистенту

Нужно давать:

- узкий scope;
- конкретные файлы;
- список инвариантов;
- список запрещённых изменений;
- критерий готовности;
- требование не делать hidden fallbacks.

Плохой промпт:
- «перепиши модуль красиво».

Хороший промпт:
- «добавь validator для checkpoint metadata в `module_target_predict`, не меняй data contract, не меняй names outputs, не трогай postprocess».

## 16. Запрещённые паттерны работы

В проекте запрещены следующие паттерны:

- писать большой модуль без предварительного scope lock;
- смешивать код, manifests и экспериментальные изменения без фиксации;
- молча менять semantics `valid`;
- молча менять `channel_semantics`;
- добавлять “на всякий случай” fallback-пути без явной документации;
- делать выводы о качестве только по одному красивому примеру;
- считать эксперимент воспроизводимым без `config_used` и manifest;
- передавать задачу дальше без handoff, если контекст нетривиален.

## 17. Короткий рабочий чеклист

Перед завершением задачи полезно пройти этот checklist.

### 17.1. До кода

- [ ] Я понимаю scope изменения.
- [ ] Я знаю, какой модуль основной.
- [ ] Я перечитал релевантные документы.
- [ ] Я понимаю, затрагивает ли это контракт.

### 17.2. После кода

- [ ] Я проверил базовую корректность.
- [ ] Я проверил manifests / summaries, если они затрагивались.
- [ ] Я не оставил скрытых fallback-магий.
- [ ] Я знаю, что именно осталось нерешённым.

### 17.3. Перед передачей дальше

- [ ] Есть понятное описание результата.
- [ ] Есть evidence проверок.
- [ ] Есть статус задачи.
- [ ] Есть следующий шаг.
- [ ] При необходимости оформлен handoff.

---

## 18. Роль документа в повседневной разработке

`DEVELOPMENT_WORKFLOW.md` должен использоваться как опорный стандарт для:

- индивидуальной разработки;
- AI-assisted coding;
- code review;
- task handoff;
- подготовки experiments;
- forensic-разбора проблемных run-ов.

Если возникает соблазн «просто быстро поменять пару файлов», не проходя этот workflow, это обычно сигнал, что именно сейчас и появляется будущий хаос.
