# python.instructions.md

## Назначение

Этот файл задаёт **общие правила Python-реализации** для проекта **«ИИ для полей»**.

Он используется при написании кода для всех модулей pipeline:

- `module_prep_data`
- `module_net_train`
- `module_target_predict`
- `module_postprocess_vectorize`
- `module_eval`

Цель файла — сделать код:

- читаемым;
- проверяемым;
- совместимым с проектными контрактами;
- удобным для тестирования, ревью и дальнейшего развития.

---

## 1. Главные принципы

### 1.1. Contract-first

Код не должен придумывать собственные правила там, где проект уже зафиксировал:

- feature contract;
- `valid` policy;
- naming contract;
- manifest policy;
- stage responsibilities.

Если реализация требует изменить контракт, сначала обновляется документация и решение, а не наоборот.

### 1.2. Явность важнее удобной магии

Код должен быть понятен по сигнатурам, данным и manifest-артефактам.

Запрещено молча:

- подменять семантику слоя;
- переставлять каналы;
- пересчитывать `valid` по другой логике без явной фиксации;
- брать другой target по неявному fallback;
- игнорировать несовместимость входов ради «пусть хоть как-то работает».

### 1.3. Маленькие функции важнее «умных комбайнов»

Функция должна делать одну инженерно понятную задачу.

Плохо:

- функция читает растр, считает индексы, собирает patch, пишет manifest и логирует итоги.

Хорошо:

- отдельно reader;
- отдельно validator;
- отдельно feature builder;
- отдельно manifest writer;
- отдельно stage runner.

## 2. Рекомендуемая структура кода

### 2.1. Что выносить в отдельные файлы

По возможности разделять:

- `cli.py` — CLI / argparse / orchestration;
- `config.py` — разбор и нормализация конфигов;
- `schemas.py` — dataclasses / typed structures / pydantic-like schemas, если используются;
- `validators.py` — явные проверки входов и контрактов;
- `io.py` — чтение/запись файлов;
- `manifest.py` — запись manifest / summary;
- `core.py` — основная логика;
- `stages/*.py` — stage-level orchestration;
- `utils.py` — только по-настоящему общие небольшие утилиты.

### 2.2. Что не смешивать

Не смешивать в одном месте:

- CLI и бизнес-логику;
- I/O и вычислительную логику;
- validators и fallback-магию;
- запись manifest и core computation;
- experiment-specific код и общий reusable код.

## 3. Сигнатуры и типизация

### 3.1. Явные входы и выходы

Функции должны по сигнатуре показывать:

- что они принимают;
- какие предположения у них есть;
- что они возвращают.

Предпочтительно использовать:

- `Path` для путей;
- `dataclass` для связных структур;
- `TypedDict` / dataclass / named structures для manifest-like объектов;
- type hints в публичных функциях.

### 3.2. Возвраты

Предпочтительно возвращать структурированные объекты, а не набор несвязных tuple-значений.

Плохо:

```python
return arr, mask, crs, transform, meta
```

Лучше:

```python
return RasterBundle(...)
```

## 4. Ошибки и исключения

### 4.1. Ошибки должны быть диагностическими

Текст ошибки должен помогать понять:

- какой контракт нарушен;
- на каком файле / артефакте;
- какое значение ожидалось;
- какое фактически получено.

### 4.2. Где надо падать явно

Нужно явно завершаться с ошибкой, если:

- нельзя надёжно интерпретировать `valid`;
- не совпадает ожидаемый `feature_mode`;
- нарушен spatial/alignment contract;
- checkpoint metadata недостаточна для predict;
- target semantics не совпадают с ожидаемыми;
- отсутствуют обязательные артефакты.

### 4.3. Чего не делать

Не использовать голые:

```python
except Exception:
    pass
```

или тихие подстановки вида:

```python
if not path.exists():
    return None
```

если отсутствие файла является контрактной ошибкой.

## 5. Логирование

### 5.1. Логи должны быть полезными

Логи должны отвечать на вопросы:

- что сейчас делает stage;
- какие входы были резолвлены;
- какие ключевые runtime-решения приняты;
- сколько объектов/патчей/тайлов обработано;
- какие warnings были обнаружены.

### 5.2. Что писать в лог обязательно

Желательно логировать:

- `run_id`;
- имя модуля и стадии;
- входные пути;
- resolved `feature_mode` / `assembled_model_input`;
- device / AMP / fallback info, если релевантно;
- summary-статистику выполнения.

### 5.3. Что не надо логировать как замену manifest

Лог не заменяет manifest.

Критичное решение должно быть восстановимо из structured output, а не только из текстовой строки лога.

## 6. Конфиги

### 6.1. Конфиг должен быть источником явных настроек

Код не должен содержать большое количество скрытых жёстко зашитых параметров, если они:

- влияют на контракт;
- влияют на сравнимость run-ов;
- должны попадать в `config_used`.

### 6.2. Нормализация конфига

После чтения конфиг должен быть приведён к нормализованному виду:

- все defaults должны быть явными;
- deprecated keys должны быть либо запрещены, либо обработаны в явном transitional adapter;
- типы должны быть приведены до начала core-логики.

---

## 7. Manifest-aware coding

### 7.1. Проектный код обязан думать про artifacts

Каждый stage должен заранее понимать:

- какие артефакты он читает;
- какие артефакты он пишет;
- что пойдёт в manifest;
- что пойдёт в summary.

### 7.2. Не делать manifest «в конце как попало»

Manifest не должен собираться из случайных остатков runtime-состояния.

Лучше:

- держать отдельные структуры `inputs_resolved`, `outputs_written`, `resolved_contract`, `runtime_info`;
- передавать их в отдельный writer.

---

## 8. Работа с путями и файлами

### 8.1. Использовать `pathlib.Path`

Во всём проекте предпочтительно использовать `Path`, а не строковые пути.

### 8.2. Не полагаться на текущую рабочую директорию

Все важные пути должны быть:

- либо абсолютными;
- либо явно резолвиться от config / repo root / module root.

### 8.3. Атомарность записи

Если пишется важный артефакт, нужно по возможности избегать частично записанных файлов без явной фиксации ошибки.

## 9. Именование

### 9.1. Имена должны быть каноническими

Использовать уже принятые имена сущностей:

- `img`
- `extent`
- `boundary`
- `distance`
- `valid`
- `meta`
- `extent_prob`
- `boundary_prob`
- `distance_pred`
- `parcel_instance`

### 9.2. Не вводить локальные синонимы без необходимости

Плохо:

- `mask_main`
- `edge_target`
- `img_stack_v2`

если в проекте уже есть канонические имена.

---

## 10. Докстринги и комментарии

### 10.1. Когда нужны комментарии

Комментарии нужны там, где без них непонятно **почему** решение именно такое.

### 10.2. Когда комментарии плохие

Плохо писать комментарии, которые дублируют код без добавления смысла.

### 10.3. Что важно объяснять

Особенно важно пояснять:

- `valid` policy;
- boundary encoding;
- compatibility checks;
- legacy adapters;
- причины отказа от «удобной» альтернативы.

---

## 11. Рефакторинг

### 11.1. Без смены поведения — значит без смены поведения

Если делается refactor, он не должен менять:

- data contract;
- manifest content;
- feature semantics;
- валидируемые ошибки;
- baseline path structure

если это не заявлено отдельно.

### 11.2. Маленькие рефакторы лучше больших переписок

Предпочтительны небольшие проверяемые изменения с тестами, чем «переписать модуль сразу целиком».

---

## 12. Чеклист перед merge

Перед merge желательно проверить:

- код не нарушает модульную границу ответственности;
- сигнатуры и названия сущностей понятны;
- обязательные ошибки действительно явные;
- manifest / summary сохраняются корректно;
- ключевая логика покрыта тестами;
- нет скрытой магии и неявных fallback-ов;
- новый код согласован с `DATA_CONTRACT.md` и модульным ТЗ.
