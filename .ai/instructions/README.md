# .ai/instructions/README.md

## Назначение

Эта папка хранит **прикладные инструкции для разработки** проекта **«ИИ для полей»**.

Её задача — помочь человеку и ИИ-ассистенту писать код так, чтобы он:

- не противоречил `main_tech.md` и модульным ТЗ;
- соблюдал `DATA_CONTRACT.md`, `MANIFEST_SCHEMAS.md`, `REPO_CONVENTIONS.md`, `TESTING_STRATEGY.md`, `EXPERIMENT_TRACKING.md` и `DECISIONS.md`;
- не ломал feature contract, `valid` policy, manifest-first policy и воспроизводимость run-ов;
- оставался удобным для ревью, дебага и дальнейшего развития.

Эти файлы **не заменяют** ТЗ модулей и не вводят новый контракт данных.
Они задают **практические правила реализации**: как писать код аккуратно внутри уже принятой архитектуры.

---

## Состав папки

### `python.instructions.md`
Правила общего Python-кодинга для проекта:

- как строить модули и функции;
- как разделять CLI, I/O, core-логику, validators и manifests;
- как писать ошибки, логи, конфиги и типизацию;
- чего нельзя делать в коде.

### `gdal_raster.instructions.md`
Правила безопасной работы с GeoTIFF, CRS, transform, alignment, NoData и `valid`.

### `torch.instructions.md`
Правила реализации train/predict логики на PyTorch с учётом:

- `raw8_valid` / `raw8_idx3_valid`;
- двойной роли `valid`;
- checkpoint-driven predict;
- AMP / device / reproducibility policy.

### `testing.instructions.md`
Практические инструкции по реализации тестов: unit, integration, smoke, golden, negative-path.

---

## Как использовать

### Для человека-разработчика

Использовать эти файлы как короткие инженерные правила перед написанием кода и перед ревью.

### Для ИИ-ассистента

Подавать нужный instruction-файл вместе с конкретной задачей.

Примеры:

- если пишется dataset reader или validator — использовать `python.instructions.md` + `gdal_raster.instructions.md`;
- если пишется train loop, checkpoint metadata или predict adapter — использовать `python.instructions.md` + `torch.instructions.md`;
- если добавляются тесты — использовать `testing.instructions.md` вместе с релевантным технологическим файлом.

## Приоритет источников

Если instruction-файл конфликтует с основными документами проекта, действует такой приоритет:

1. архитектурный freeze / принятые решения (`DECISIONS.md`);
2. `main_tech.md`;
3. модульные ТЗ (`module_*.md`);
4. `DATA_CONTRACT.md`;
5. `MANIFEST_SCHEMAS.md`;
6. `REPO_CONVENTIONS.md`;
7. `TESTING_STRATEGY.md`;
8. `EXPERIMENT_TRACKING.md`;
9. `.ai/instructions/*`.

То есть инструкции — это **уровень реализации**, а не источник права менять проектный контракт.

---

## Общие правила для всех instruction-файлов

Любая реализация в проекте должна:

- быть **contract-first**, а не «code-first»;
- быть **manifest-aware**;
- не терять provenance;
- не скрывать критичную логику за fallback-магией;
- падать с явной ошибкой там, где контракт нельзя восстановить надёжно;
- не смешивать обязанности соседних модулей pipeline;
- учитывать, что `valid` — это одновременно служебная mask и входной канал модели;
- сохранять воспроизводимость run-ов как часть baseline engineering policy.

---

## Когда нужно добавлять новые instruction-файлы

Новый файл имеет смысл добавлять, если:

- в проекте появляется новый стабильный технологический слой;
- для него нужны повторяемые инженерные правила;
- этих правил слишком много для одного общего файла;
- они реально уменьшают количество ошибок при кодинге и ревью.

Не стоит плодить instruction-файлы для каждой мелкой детали.

---

## Что сюда не выносить

Сюда **не надо** выносить:

- полное ТЗ модуля;
- исследовательские заметки;
- результат конкретного эксперимента;
- временные workaround-решения;
- куски чата без нормализации в инженерную форму.

---

## Минимальный режим применения

Если нет времени подключать все инструкции, минимально рекомендуется:

- всегда подключать `python.instructions.md`;
- для геоданных — `gdal_raster.instructions.md`;
- для train/predict — `torch.instructions.md`;
- перед merge — `testing.instructions.md`.
