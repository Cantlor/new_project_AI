Общее ТЗ v1 — ИИ для полей

1. Назначение системы

Система предназначена для автоматического выделения и векторизации сельскохозяйственных полей по спутниковым мультиспектральным снимкам высокого разрешения.

Целевой сценарий:

на вход подается 8-бэндовый GeoTIFF с пространственным разрешением около 3 м/пиксель;
снимок может иметь произвольные метаданные: собственный CRS, dtype, значение NoData, диапазоны значений пикселей, геопривязку, тайлинг, компрессию;
система приводит данные к внутреннему рабочему контракту;
далее выполняются подготовка признаков, инференс модели, сохранение растровых предсказаний и последующий постпроцессинг с векторизацией;
на выходе пользователь получает карты extent / boundary / distance, итоговый векторный слой полей и сопровождающие отчеты / манифесты.

2. Цели проекта
Получать устойчивое разделение соседних полей, включая сложные внутренние границы.
Минимизировать ложные слияния и ложные разделения.
Корректно обрабатывать NoData / invalid зоны и не создавать на них ложные предсказания.
Сохранять геопривязку и воспроизводимость на всех стадиях пайплайна.
Обеспечить единый контракт между обучением и предиктом.
Обеспечить переносимость пайплайна между разным железом без ручного переписывания логики.

3. Бизнес-результат / пользовательский результат
Пользователь должен иметь возможность:

обучить модель на подготовленном датасете;
применить обученную модель к новым растрам без ручной перенастройки кода;
получить понятный и воспроизводимый результат в виде геопривязанного векторного слоя полей;
сравнивать разные режимы признаков и разные версии моделей по единым метрикам;
запускать проект на доступной среде исполнения с автоматической адаптацией под устройство и память.

4. Основная архитектурная идея

Проект строится как pipeline из отдельных модулей:

module_prep_data — подготовка train-ready датасета;
module_net_train — обучение и оценка нейросети;
module_target_predict — tiled inference по новым растрам;
module_postprocess_vectorize — постпроцессинг и векторизация;
module_eval — единая система оценки качества.

Базовая целевая ML-постановка:

не бинарная сегментация field / background;
а boundary-aware multitask-подход с предсказанием:
extent — протяженность / маска поля;
boundary — карта границ;
distance — карта расстояния до ближайшей границы.

5. Входные данные
5.1. Основной вход
GeoTIFF с 8 спектральными каналами;
пространственное разрешение целевого кейса: около 3 м/пиксель;
размер растра произвольный;
возможны разные CRS, NoData, dtype, диапазоны значений.
5.2. Разметка для обучения
векторные полигоны полей;
разметка должна быть геометрически согласована с растром;
внутренние границы считаются критически важной частью ground truth.
5.3. Дополнительные данные
AOI — рекомендованный входной объект, если доступен;
в дальнейшем допускается поддержка дополнительных масок / подсказок, но не в первой версии ТЗ.

6. Внутренний рабочий контракт данных
6.1. Feature modes

Официально поддерживаются только два финальных режима модельного входа:

raw8_valid
raw8_idx3_valid

Где:

raw8_valid = 8 спектральных каналов + valid
raw8_idx3_valid = 8 спектральных каналов + NDVI + SAVI + NDWI + valid

Итого:

raw8_valid = 9 каналов
raw8_idx3_valid = 12 каналов

valid является не только служебной маской, но и дополнительным входным каналом модели.

6.2. Train target contract

Для каждого patch / sample должны существовать:

img
valid
extent
boundary
distance
meta
6.3. Политика NoData / valid
valid вычисляется до любой замены значений NoData;
NoData не должен смешиваться с обычным фоном;
invalid зоны исключаются из loss через ignore policy;
на инференсе invalid зоны подавляются отдельно;
valid используется одновременно в двух ролях:
как служебная valid-mask для ignore / masking / postprocess;
как дополнительный входной канал модели.

7. Политика нормализации и приведения растров
7.1. Что обязательно приводить к единому контракту

Обязательно приводятся:

число и порядок каналов;
правила интерпретации NoData;
spatial alignment с разметкой;
рабочая форма тензора;
нормализация / масштабирование значений для модели;
единый internal manifest.
7.2. Что не обязательно унифицировать физически на диске

Не требуется принудительно переписывать каждый входной растр в один и тот же dtype, если:

сохраняется числовая точность;
корректно обрабатывается NoData;
на вход модели данные все равно подаются в одном внутреннем формате, например float32 tensor;
не возникает переполнения, потери динамического диапазона или ошибок масштабирования.
7.3. Финальная политика нормализации v1
valid вычисляется до любых преобразований;
далее данные приводятся к float32;
используется per-band robust percentile normalization, рассчитанная только по valid train pixels;
базовый диапазон clipping: p0.5 / p99.5;
после clipping выполняется scaling в [0, 1];
на predict применяются те же train-derived statistics, без их пересчета на новом растре.
7.4. Что критически важно
одинаковая логика препроцессинга на train и predict;
явное хранение статистик / масштабирования в конфиге, чекпоинте или манифесте;
отсутствие тихих преобразований dtype / range без следа в metadata.

8. Политика AOI

AOI не является строго обязательным входом, однако в первой стабильной версии считается рекомендуемым режимом работы.

Принятое решение:

если AOI доступен, система по умолчанию должна уметь использовать его для ограничения рабочей области на этапах подготовки данных, инференса и / или постпроцессинга;
если AOI отсутствует, система должна оставаться работоспособной;
применение AOI должно быть конфигурируемым;
при использовании AOI требуется аккуратная buffer-политика, чтобы не терять полезный контекст у границ AOI.

Базовая buffer-политика v1:

baseline buffer вокруг AOI: 30 м.

Практический смысл AOI:

снижение доли нерелевантного фона;
сокращение ложных срабатываний вне интересующей территории;
ускорение пайплайна;
упрощение постпроцессинга и итоговой векторизации.

9. Требования к модели

Целевая нейросеть первой основной версии:

гибридная edge-aware multitask encoder-decoder сеть;
три головы: extent, boundary, distance;
deep supervision;
boundary-aware loss;
поддержка режимов raw8_valid и raw8_idx3_valid.

Базовая архитектурная схема v1:

CNN-first residual encoder;
lightweight global-context / transformer-like bottleneck;
multi-scale fusion;
edge-aware blocks;
refine module;
heads = extent / boundary / distance.

Подробная архитектура фиксируется в отдельном ТЗ для module_net_train.

10. Требования к подготовке данных

module_prep_data должен:

поддерживать patch sizes 256 / 384 / 512;
использовать strategic sampling, а не только чистый random;
строить boundary linework-faithful от реальных polygon boundaries;
экспортировать обязательные слои:
img
extent
boundary
distance
valid
meta
сохранять boundary_raw как обязательный промежуточный диагностический артефакт.

Базовые решения v1:

baseline patch size = 512
boundary encoding = background / skeleton / buffer
distance target = unsigned distance-to-nearest-boundary
экспортный формат patch-level raster artifacts = GeoTIFF + meta.json
11. Требования к обучению

module_net_train должен:

обучать multitask-модель по heads extent / boundary / distance;
строго соблюдать valid / ignore policy;
поддерживать оба feature mode без ручного переписывания пайплайна;
сохранять checkpoint metadata, достаточную для полностью автоматического predict;
обеспечивать воспроизводимый experiment tracking.

Базовые решения v1:

optimizer: AdamW
scheduler: cosine with warmup
AMP включается автоматически, если это безопасно и поддерживается средой;
стартовые веса losses:
extent = 1.0
boundary = 2.5
distance = 1.0
лучшая модель выбирается по composite metric:
0.6 *boundary_F1 + 0.4* extent_F1

Первые официальные сравнения:

raw8_valid
raw8_idx3_valid
12. Требования к предикту

Система должна:

автоматически определять ожидаемое число входных каналов модели;
автоматически восстанавливать нужный feature contract из checkpoint metadata;
при необходимости автоматически строить derived indices из исходного 8-band растра;
автоматически добавлять valid как входной канал, если модель этого требует;
выполнять tiled inference с overlap;
сохранять геопривязку выходных растров;
формировать manifest предикта.

Базовые решения v1:

checkpoint-driven inference;
если checkpoint ожидает 9 каналов — собирается raw8_valid;
если checkpoint ожидает 12 каналов — собирается raw8_idx3_valid;
baseline tile size = 512
baseline overlap = 25%
baseline blending = Gaussian
invalid-only tiles должны пропускаться.

Обязательные raster outputs v1:

extent_prob.tif
boundary_prob.tif
distance_pred.tif
valid.tif

Predict не должен выполнять thresholding, watershed или polygonization внутри базового inference stage.

13. Требования к постпроцессингу

module_postprocess_vectorize должен:

принимать extent_prob, boundary_prob, distance_pred, valid;
выполнять suppression по valid и при наличии по AOI;
выполнять калиброванное thresholding;
использовать boundary-aware восстановление отдельных parcel instances;
строить итоговый parcel_instance.tif;
выполнять polygonization;
выполнять консервативную топологическую очистку;
экспортировать итоговый векторный слой.

Базовый pipeline v1:

valid / AOI suppression
validation-calibrated thresholding
hybrid boundary repair
marker generation = extent-core ∩ low-boundary ∩ high-distance
marker-controlled watershed
region filtering / constrained merge
parcel_instance.tif
polygonization
conservative topology cleanup
export

Обязательные outputs:

parcel_instance.tif
итоговый векторный слой в формате GPKG
polygon_confidence как rule-based QC score

Shapefile допускается только как optional export.

14. Требования к оценке качества

Оценка качества должна вестись не только по pixel-метрикам.

Обязательные группы метрик:

global / pixel metrics;
boundary metrics;
object / structure metrics.

Минимально обязательные метрики v1:

extent: IoU / F1 / Precision / Recall
boundary: F1 / Precision / Recall / BDE
object / structure: GOC / GUC / GTC

Также должны поддерживаться отчеты по сложным кейсам:

простые крупные поля;
мелкие / фрагментированные поля;
слабые внутренние границы;
области рядом с дорогами, зданиями, водой;
near-invalid / near-NoData зоны.

Дополнительно обязательны:

bucketed evaluation;
error taxonomy;
multi-run comparison.
15. Hardware-adaptive runtime policy

Все вычислительно тяжелые модули проекта должны автоматически адаптироваться к доступной среде выполнения.

Базовый приоритет устройств:

CUDA -> MPS -> CPU

Общие правила:

AMP включается автоматически там, где это безопасно и поддерживается средой;
при нехватке памяти модуль должен деградировать по runtime-параметрам, а не ломать feature contract, valid-policy или архитектурную семантику;
порядок деградации должен быть воспроизводимым и явно отражаться в logs / manifests;
пользователь может вручную переопределить device/runtime settings, но default path должен быть automatic and hardware-aware.

Общая политика деградации under OOM:

сначала уменьшать tile_batch_size или batch_size;
затем уменьшать вторичные runtime-параметры;
затем включать или увеличивать accumulation там, где это применимо;
только после этого переходить к более медленному, но корректному режиму исполнения.

Нельзя ради экономии памяти:

убирать valid из входного контракта модели;
ломать ignore / valid handling;
менять число каналов без явного изменения feature mode;
менять train / predict preprocessing logic;
отключать heads multitask-модели без явного нового режима.
16. Нефункциональные требования
воспроизводимость запусков;
manifest-first дизайн;
конфигурируемость через YAML;
минимизация скрытых допущений;
прозрачная диагностика причин деградации качества;
сохранение оригинальных исходных растров без их обязательной перезаписи;
совместимость с hardware-adaptive режимом;
явное логирование resolved runtime environment.
17. Что не входит в первую версию

Не включается в первую стабильную версию:

contour-first / vector-first нейросеть как основной production path;
foundation model как обязательная зависимость;
объединение parcel delineation и crop-type classification в одном обучении;
поддержка произвольного числа сенсоров без явного контракта;
обещание одинаково качественной работы модели на разных источниках данных без отдельной проверки transfer / generalization.
18. Артефакты каждого run

Каждый run любого модуля должен порождать:

config_used.*
manifest.*
summary.*
диагностические логи
версионируемые outputs

Если модуль runtime-sensitive, в manifest дополнительно рекомендуется сохранять:

device_requested
device_resolved
amp_requested
amp_used
oom_fallbacks_applied
19. Критерии приемки общего дизайна

Общий дизайн считается принятым, если:

Между модулями нет неоднозначности во входах / выходах.
Train и predict используют один и тот же внутренний контракт признаков.
NoData / valid политика зафиксирована и воспроизводима.
valid используется и как служебная маска, и как часть модельного входа.
Есть возможность обучить модель и применить ее к новому растру без ручного переписывания пайплайна.
Есть единая схема оценки качества и сравнения экспериментов.
Система остается работоспособной на разных средах исполнения за счет hardware-adaptive policy.
