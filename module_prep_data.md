# ТЗ `module_prep_data` v1

## 1. Назначение модуля

`module_prep_data` отвечает за подготовку train-ready датасета для задачи выделения и разделения сельскохозяйственных полей по 8-бэндовым спутниковым снимкам высокого разрешения.

Модуль не должен рассматриваться как простой «резак патчей». Его задача — сформировать корректный и воспроизводимый ML-контракт между исходными геоданными и последующими модулями обучения/инференса.

## 2. Границы ответственности

### 2.1. Модуль обязан

* читать исходные растры и векторную разметку;
* проверять и приводить пространственную согласованность данных;
* интерпретировать NoData/valid;
* при необходимости применять AOI;
* формировать feature stack в режимах `raw8` и `raw8_idx3`;
* строить train targets: `extent`, `boundary`, `distance`, `valid`;
* выполнять sampling патчей;
* экспортировать train/val/test датасет по единому контракту;
* сохранять manifests, summaries и диагностику.
* формировать metadata, достаточную для однозначного восстановления downstream model input contract;
* обеспечивать совместимость экспортируемого датасета с последующими модулями без неявных догадок о числе каналов, порядке каналов и роли valid-mask.

### 2.2. Модуль не обязан

* обучать модель;
* выполнять inference модели;
* делать финальную векторизацию результатов;
* рассчитывать итоговые production-метрики качества модели.

## 3. Входные данные

### 3.1. Основные входы

1. Мультиспектральный raster в формате GeoTIFF:

   * 8 каналов;
   * разрешение целевого кейса около 3 м/px;
   * произвольный исходный dtype;
   * произвольный исходный CRS;
   * собственный NoData или отсутствие корректно заданного NoData.
2. Векторная разметка полей:

   * polygons / multipolygons;
   * согласуемый с растром CRS;
   * опциональное поле идентификатора.
3. AOI (опционально, но рекомендуется):

   * polygon / multipolygon;
   * может использоваться как ограничитель области обработки.

### 3.2. Входные допущения

* Оригинальные входные файлы не переписываются и не модифицируются.
* Все преобразования выполняются только во внутреннем рабочем слое `output/work/...`.
* Если метаданные неконсистентны, модуль обязан либо восстановить корректную интерпретацию, либо завершиться с явной диагностикой.

## 4. Выходные данные

### 4.1. Основной результат

Подготовленный датасет для `module_net_train` со следующей структурой на каждый split:

* `img/`
* `extent/`
* `boundary/`
* `distance/`
* `valid/`
* `meta/`

### 4.2. Сопутствующие артефакты

* `check_inputs_manifest.json`
* `aoi_manifest.json`
* `features_manifest.json`
* `patches_manifest.json`
* `split_manifest.json`
* `summary.json`
* выборочные sanity-изображения/оверлеи
* статистика sampling и отбраковок

## 5. Общий pipeline модуля

Модуль должен быть организован как последовательность стадий:

1. `01_check_inputs`
2. `02_prepare_spatial_context`
3. `03_prepare_features`
4. `04_prepare_targets`
5. `05_make_patches`
6. `06_split_dataset`
7. `07_validate_outputs`

Каждая стадия должна быть перезапускаемой и иметь собственный manifest.

## 6. Политика пространственного согласования

### 6.1. CRS и alignment

* Растр считается пространственным эталоном сетки.
* Векторная разметка и AOI должны приводиться к CRS растра.
* Все target raster layers должны быть выровнены по grid растра.
* Не допускается тихий half-pixel drift.

### 6.2. AOI policy

Если AOI предоставлен:

* модуль должен уметь ограничивать им рабочую область;
* по умолчанию должен использоваться буфер вокруг AOI, чтобы не терять контекст по краям;
* buffer policy должна быть конфигурируемой;
* обрезка по AOI не должна уничтожать полезные boundary-сигналы на краях.

Если AOI отсутствует:

* модуль должен оставаться полностью работоспособным.

## 7. Политика NoData и valid

### 7.1. Принципы

* `valid` вычисляется до любой замены NoData на fill value.
* NoData не должен интерпретироваться как обычный фон.
* `valid` хранится как отдельный слой.
* invalid пиксели должны исключаться из loss в downstream модулях.
* `valid` используется не только как служебный слой, но и как часть сквозного train/predict контракта.
* Сам `module_prep_data` экспортирует `valid` отдельным raster-слоем и не обязан физически вшивать его внутрь `img`.
* Downstream-модули должны иметь возможность однозначно собрать модельный вход из feature stack и слоя `valid`.

### 7.2. Источники valid-mask

Приоритет интерпретации valid:

1. dataset mask / internal mask / sidecar mask, если она существует;
2. явно заданный nodata в metadata;
3. пользовательское правило из конфигурации, если metadata недостаточна;
4. явная ошибка, если корректная интерпретация невозможна.

### 7.3. Внутреннее представление

* `valid` хранится как 0/1 raster.
* Во внутренних пайплайнах допускается промежуточная GDAL-style valid-mask интерпретация.
* Любая замена invalid значений на 0 или иное fill value допустима только после фиксации `valid`.

## 8. Политика dtype и значений пикселей

### 8.1. Что не требуется

* Не требуется переписывать все оригинальные входные GeoTIFF в единый disk dtype.
* Не требуется физически менять оригинальные NoData значения в исходниках.

### 8.2. Что требуется

* На этапе формирования train-ready sample данные должны приводиться к единому внутреннему представлению для модели.
* Базовое внутреннее представление входных тензоров: `float32`.
* Должна существовать единая политика масштабирования / нормализации.
* Любая нормализация должна быть одинаковой для train и predict.

### 8.3. Ограничения

* Запрещены скрытые преобразования, способные повредить динамический диапазон.
* Запрещено терять информацию из-за неявного кастинга без записи этого факта в metadata/manifests.

## 9. Feature modes

### 9.1. Официальные режимы

Поддерживаются только два режима:

* `raw8`
* `raw8_idx3`

### 9.2. Режим `raw8`

* Используются 8 исходных спектральных каналов в фиксированном порядке.
* Порядок каналов должен быть явно зафиксирован в конфиге и manifest.

### 9.3. Режим `raw8_idx3`

* Используются те же 8 исходных каналов;
* дополнительно рассчитываются 3 производных индекса.

### 9.4. Зафиксированный состав индексов для v1

Для режима `raw8_idx3` в первой версии фиксируются:

* NDVI
* SAVI
* NDWI

### 9.5. Связь feature modes с downstream model input contract

`module_prep_data` формирует feature stack в режимах `raw8` и `raw8_idx3`, а слой `valid` сохраняется отдельно.

При этом модуль должен экспортировать metadata так, чтобы downstream-модули могли без догадок восстановить финальный модельный вход:

* `raw8_valid = raw8 + valid`
* `raw8_idx3_valid = raw8_idx3 + valid`

Соответственно:

* `raw8` содержит 8 feature-каналов, а финальный model input после добавления `valid` содержит 9 каналов;
* `raw8_idx3` содержит 11 feature-каналов, а финальный model input после добавления `valid` содержит 12 каналов.

Это правило должно рассматриваться как часть общего сквозного контракта проекта, но не означает, что `module_prep_data` обязан физически объединять `img` и `valid` в один экспортный файл.

## 10. Политика нормализации

### 10.1. Требования

Модуль должен поддерживать конфигурируемую схему нормализации, общую для train и predict.

### 10.2. Базовый рекомендуемый вариант

Для v1 рекомендуется использовать per-channel robust statistics на train split:

* например percentile clipping;
* затем масштабирование в стабильный диапазон;
* затем преобразование в `float32`.
Для baseline v1 этот рекомендуемый вариант считается принятым по умолчанию, если конфиг явно не задает иной совместимый режим:

* `valid` вычисляется до любых числовых преобразований;
* входные данные приводятся к `float32`;
* статистики считаются только по valid-пикселям train split;
* используется per-band robust percentile normalization;
* baseline clipping фиксируется как `0.5 / 99.5`;
* далее выполняется масштабирование в диапазон `[0, 1]`;
* те же зафиксированные статистики затем используются и на predict.

### 10.3. Запрещается

* вычислять несовместимые статистики отдельно для train и predict без явной фиксации;
* использовать разные правила препроцессинга для разных этапов одного и того же feature mode.

## 11. Генерация targets

### 11.1. `extent`

* Бинарная карта принадлежности полю.
* Должна поддерживать ignore policy около invalid-зон и при необходимости в окрестности boundary.
* Внутреннее хранение: 0/1/255, где 255 = ignore.

### 11.2. `boundary`

* Должна строиться из реального polygon linework, а не только из morphological gradient extent-mask.
* Внутренние границы между соседними полигонами считаются обязательными.
* Для первой сильной версии модуля принимается boundary-aware кодирование в формате `skeleton / buffer / background`.
* Базовая целевая семантика кодов:

  * `0` — background
  * `1` — skeleton boundary
  * `2` — buffer / ignore-like belt around boundary
* Ширина buffer-зоны должна быть конфигурируемой, но иметь безопасный baseline.
* Дополнительно модуль может сохранять диагностический `boundary_raw` для визуальной и forensic-проверки.
Для baseline v1 `boundary_raw` сохраняется по умолчанию всегда и рассматривается как обязательный диагностический артефакт, даже если он не входит в основной train/export contract.

### 11.3. `distance`

* Должна формироваться как карта расстояния до ближайшей границы.
* Для v1 принимается `unsigned euclidean distance-to-nearest-boundary`.
* На самой границе значение должно быть равно 0.
* Остальные пиксели должны хранить неотрицательное расстояние до ближайшей boundary.
* Диапазон/клиппинг и схема нормализации должны быть едиными и явно зафиксированными в конфиге.
* Signed-представление в первую стабильную версию не включается.

### 11.4. `valid`

* Формируется отдельно и сохраняется как самостоятельный таргетный/служебный слой.

## 12. Patch generation

### 12.1. Общие требования

* Патчи должны извлекаться по grid растра.
* Размер патча и stride должны задаваться конфигом.
* В первой версии модуль обязан поддерживать как минимум три patch size режима: `256`, `384`, `512`.
* Допускается запуск как в режиме одного выбранного patch size, так и в режиме batch-generation для нескольких размеров подряд.
* Должны поддерживаться режимы centered sampling, boundary sampling и negative sampling.

### 12.2. Требования к содержанию патча

Для каждого патча должны вычисляться и сохраняться:

* `valid_ratio`
* `mask_ratio`
* `edge_ratio`
* `source_polygon_count`
* `sampling_reason`
* `feature_mode`
* `xoff`, `yoff`

### 12.3. Политика отбраковки

Патч может быть отбракован, если:

* слишком мал `valid_ratio`;
* отсутствует полезный target signal для выбранного sampling mode;
* патч является дубликатом или почти дубликатом по заданным правилам.

Причина отбраковки должна логироваться в summary.

## 13. Sampling strategy

### 13.1. Основной принцип

Sampling должен быть стратегическим, а не только случайным.

### 13.2. Обязательные классы патчей

Для v1 должны поддерживаться как минимум:

* `center_positive`
* `boundary_positive`
* `hard_negative`
* `near_invalid`

### 13.3. Желательные дополнительные классы

* `lookalike_neighbor`
* `multi_polygon_conflict`
* `thin_boundary_case`

### 13.4. Цель sampling

* насыщать датасет сложными boundary-примерами;
* уменьшать долю «простых» центральных патчей;
* сохранять достаточное количество hard negatives;
* повышать устойчивость модели у valid/NoData и boundary-зон.

## 14. Split policy

### 14.1. Требования

* split должен быть конфигурируемым;
* split должен быть воспроизводимым;
* запрещена утечка между train/val/test через дубликаты и near-duplicate patches;
* предпочтителен spatially aware split.

### 14.2. Минимум для metadata

Для split manifest должны сохраняться:

* seed
* feature_mode
* patch_size
* counts by sampling class
* counts by split
* counts of rejected samples

## 15. Экспортный контракт

### 15.1. Формат

Экспорт должен быть простым, стабильным и пригодным для чтения из `module_net_train` без догадок.

Для v1 основной экспортный формат принимается как GeoTIFF для растровых слоев и JSON для metadata.
Это решение выбрано в том числе для удобной визуальной проверки и аудита данных в ArcGIS.

### 15.2. Обязательные подпапки

На каждый split:

* `img/`
* `extent/`
* `boundary/`
* `distance/`
* `valid/`
* `meta/`

### 15.3. Требования к именованию

* Все связанные файлы одного sample должны иметь общий sample id.
* Имена не должны зависеть от скрытых внутренних индексов без отражения в metadata.
* Metadata одного sample должна явно описывать:
  * `feature_mode`;
  * `feature_channel_count_before_valid`;
  * `model_input_channel_count_after_valid`;
  * `channel_names`;
  * наличие и семантику слоя `valid`.
* Экспортный контракт должен позволять downstream-модулю восстановить ожидаемый model input contract без fallback-магии.

## 16. Валидация и sanity checks

### 16.1. Проверки после генерации

Модуль должен иметь встроенные проверки:

* совпадение shapes у всех слоев одного sample;
* корректность допустимых значений таргетов;
* отсутствие неожиданных пустых/битых файлов;
* согласованность valid и NoData policy;
* корректность boundary coverage;
* корректность distance map относительно boundary.
* корректность metadata, описывающей downstream model input contract;
* согласованность `feature_mode`, числа feature-каналов и ожидаемого числа model input channels после добавления `valid`;
* наличие достаточной информации для восстановления train/predict preprocessing без неявных допущений.

### 16.2. Визуальные sanity artifacts

Должны сохраняться:

* overlay image + extent;
* overlay image + boundary;
* overlay image + valid;
* несколько сложных патчей по каждой sampling class.

## 17. Конфигурирование

### 17.1. Конфиг обязан задавать

* пути ко входам;
* рабочие директории;
* feature_mode;
* band order / band mapping;
* nodata/valid policy;
* AOI policy и buffer;
* patch_size;
* sampling configuration;
* split configuration;
* export format settings.
* политику нормализации и источник статистик;
* политику интерпретации valid / nodata при недостаточных metadata;
* правила описания channel semantics в export metadata.

### 17.2. Manifest-first policy

Каждый stage-run должен сохранять:

* `config_used.*`
* stage manifest
* summary
* warnings/errors

## 18. Ошибки и диагностика

Модуль обязан завершаться с явной ошибкой, если:

* невозможно однозначно интерпретировать band order;
* raster/vector spatial alignment некорректен;
* NoData/valid policy не может быть восстановлена безопасно;
* feature mode требует индексы, которые нельзя корректно посчитать;
* split/export нарушают внутренний контракт.

## 19. Критерии приемки модуля

Модуль считается готовым для интеграции, если:

1. Из входного 8-band GeoTIFF и векторной разметки формируется воспроизводимый train-ready датасет.
2. `raw8` и `raw8_idx3` поддерживаются одним и тем же пайплайном.
3. `valid`, `extent`, `boundary`, `distance` сохраняются как отдельные согласованные слои.
4. Внутренние границы не теряются при генерации `boundary`.
5. AOI, если задан, корректно ограничивает рабочую область без разрушения полезного контекста.
6. Все стадии имеют manifests и могут быть переиспользованы без ручных правок кода.
7. Экспортный контракт достаточно строг, чтобы `module_net_train` мог читать датасет без fallback-магии.
8. Экспортируемые данные и metadata позволяют downstream-модулям однозначно восстановить финальный model input contract (`raw8_valid` или `raw8_idx3_valid`) без скрытых предположений.

## 20. Зафиксированные baseline-решения перед кодингом

1. Базовый patch size для первого сильного baseline: `512`.
2. Модуль обязан поддерживать также `256` и `384`, но первый официальный baseline запускается на `512`.
3. Базовая схема нормализации для v1:

   * вычисление `valid` до любых числовых преобразований;
   * приведение входа модели к `float32`;
   * `per-band robust percentile normalization` по статистикам train split, рассчитанным только по valid пикселям;
   * clipping по перцентилям `0.5 / 99.5`;
   * последующее масштабирование в диапазон `[0, 1]`;
   * одни и те же зафиксированные статистики используются и на train, и на predict.
4. Если исходные данные уже приведены к физически осмысленному радиометрическому представлению (например, reflectance) и это можно сделать надежно, такое представление считается предпочтительным входом до этапа robust normalization.
5. Базовый AOI buffer для v1 фиксируется в метрах и по умолчанию принимается равным `30 м`.
6. AOI buffer должен применяться как контекстный внешний запас вокруг AOI, а не как обрезание полезной информации.
7. Диагностический слой `boundary_raw` сохраняется по умолчанию всегда.
8. Помимо `boundary_raw`, модуль по умолчанию сохраняет preview/sanity overlays для image+extent, image+boundary, image+valid.
9. `valid` по умолчанию сохраняется как отдельный обязательный слой и одновременно считается частью сквозного downstream model input contract.
10. Export metadata по умолчанию должны содержать информацию, достаточную для восстановления финального числа каналов модели после добавления `valid`.
