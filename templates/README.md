# templates/

Эта папка содержит **шаблоны служебных файлов проекта `ИИ для полей`**.

Назначение шаблонов:

- ускорять создание новых run-артефактов;
- удерживать единый стиль manifests / summaries / reports / handoff-документов;
- уменьшать риск потери provenance, contract metadata и runtime context;
- помогать людям и ИИ-ассистентам создавать файлы в одном формате.

## Состав

- `manifest.template.json` — базовый шаблон manifest-файла любого модуля.
- `summary.template.json` — базовый шаблон краткого summary-файла.
- `config_used.template.yaml` — шаблон зафиксированного конфига запуска.
- `run_report.template.md` — шаблон человекочитаемого технического отчёта по run.
- `handoff.template.md` — шаблон handoff-документа для продолжения работы в новом чате/другим исполнителем.

## Как использовать

1. Скопировать подходящий шаблон в run-директорию.
2. Переименовать файл под конкретный модуль или запуск.
3. Заполнить все обязательные поля.
4. Не оставлять критичные placeholders в production-like run.

## Общие правила

- `run_id` обязателен.
- `data_contract_version` обязателен.
- `source_run_ids`, `source_manifest_paths`, `config_used_path`, `resolved_contract`, `runtime` и `diagnostics` нельзя терять, если они реально влияли на результат.
- Если поле не применимо, лучше явно указать `null`, чем молча опустить критичную информацию.
- Если модуль влияет на downstream-совместимость, в артефактах должен быть восстановим `feature_mode`, `assembled_model_input`, `valid_policy` и источник normalization stats.

## Что не делать

- Не использовать шаблоны как замену реальным manifests.
- Не оставлять фиктивные значения вроде `TBD` в финальных run-артефактах.
- Не скрывать runtime fallback-решения, даже если они кажутся «временными».
