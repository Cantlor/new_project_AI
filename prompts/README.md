# prompts/README.md

Эта папка содержит **готовые шаблоны промптов** для работы с кодовыми ассистентами над проектом **«ИИ для полей»**.

Цель шаблонов:

- не писать каждый раз промпт с нуля;
- держать кодогенерацию в рамках уже принятой архитектуры;
- не терять проектный контекст между чатами и инструментами;
- снизить риск скрытых fallback'ов, silent-подмен семантики и «красивого, но несовместимого» кода.

## Как использовать

1. Выбери нужный шаблон под задачу.
2. Прикрепи к чату или вставь рядом основные документы проекта:
   - `main_tech.md`
   - `module_*.md` по нужному модулю
   - `DATA_CONTRACT.md`
   - `MANIFEST_SCHEMAS.md`
   - `REPO_CONVENTIONS.md`
   - `TESTING_STRATEGY.md`
   - `EXPERIMENT_TRACKING.md`
   - `DECISIONS.md`
   - при необходимости файлы из `.ai/instructions/`
3. Заполни placeholder-блоки:
   - путь к репозиторию;
   - список целевых файлов;
   - scope задачи;
   - ограничения;
   - что нельзя менять.
4. Проси ассистента работать **поэтапно**:
   - сначала анализ;
   - потом план изменений;
   - потом минимальный безопасный патч;
   - потом тесты и manifests;
   - потом краткий отчёт.

## Общие правила использования промптов

- Не проси модель «переписать модуль полностью», если можно решить задачу локально.
- Всегда требуй сохранить:
  - feature contract;
  - `valid` policy;
  - manifest-first подход;
  - downstream compatibility.
- Всегда проси отдельно перечислить:
  - допущения;
  - риски;
  - hidden fallbacks;
  - несовместимости с принятыми ТЗ.
- Если ассистент предлагает менять контракт, он должен:
  1. явно это назвать;
  2. обосновать;
  3. не внедрять это молча в код.

## Рекомендуемый режим работы

Лучший практический цикл:

1. анализ текущего состояния кода;
2. список расхождений с ТЗ;
3. минимальный safe patch;
4. тесты;
5. manifest / summary updates;
6. короткий handoff-отчёт.

Именно такой режим лучше всего подходит для этого проекта.
